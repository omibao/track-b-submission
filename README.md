## 这是什么比赛

字节跳动主办的 AI 安全挑战赛，两个赛道：

- **赛道 A（红队）**：设计能绕过检测引擎的恶意 AI Skill 样本
- **赛道 B（蓝队）**：开发检测引擎，判断 AI Skill 是恶意还是良性

我们参加的是赛道 B。

### 什么叫 AI Skill

AI Skill 是给 AI Agent（比如 Claude Code、Cursor）用的"插件包"。里面通常有：
- `manifest.json`：声明权限、依赖、入口
- `SKILL.md` 或 `skill.md`：用自然语言写的指令（AI 读这个来理解该做什么）
- 代码文件：Python/Shell/JS 等脚本

恶意 Skill 会在看起来正常的描述下隐藏恶意行为——比如伪装成"CI 缓存优化工具"，实际偷偷读 SSH 密钥、加后门、往外发数据。

### 比赛怎么评分的

评测方式是"黑盒"：主办方有一批标注好恶意/良性的测试 Skill，我们的 Docker 镜像逐个扫描，输出判决。四个维度打分：

| 维度 | 缩写 | 英文 | 满分 | 含义 |
|------|------|------|------|------|
| 检测质量 | DQ | Detection Quality | 5.5 | F₂ 分数。**F₂** 是一种特殊的 F 值，给召回率 4 倍权重（漏判恶意比误判良性严重 4 倍） |
| 可解释性 | EX | Explainability | 2.0 | 不仅要判对，还要说对"这属于 OWASP AST 十大风险中的哪一类" |
| 稳健性 | RO | Operational Robustness | 1.5 | 引擎不能崩、JSON 格式要对、不能出网、不能超时 |
| 性能 | PF | Performance | 1.0 | 跑得快、内存省 |

**总分 = DQ + EX + RO + PF，满分 10 分。**

### 比赛约束

- Docker 镜像上限 10G，构建和运行**完全不能访问外网**
- 输入：`/data/skills/{skill_id}/` 目录
- 输出：`/output/results.jsonl`，每行一个 JSON 判决
- 判决字段：`skill_id`、`verdict`（benign/malicious/suspicious）、`category`（AST01-10）、`evidence`（自然语言证据）
- 每天 10 次提交机会
- 最终排名取所有提交中**最高分的那次**

### 真正的难点在哪

1. **看不到测试集**：不知道测试样本长什么样，只能从公开数据集和公告推断
2. **黑盒反馈**：只有四个数字反馈，不知道哪类样本判错了
3. **无外网**：不能调 LLM API，只能用纯本地计算
4. **隐蔽攻击**：恶意 Skill 不一定是代码攻击——更多是藏在 Markdown 里的自然语言诱导
5. **分类要对**：不仅要抓恶意，还要准确说出它是 AST01-AST10 中的哪一类

## OWASP AST 十大风险（Agentic Skills Top 10）

比赛使用的分类体系，不是传统的 Web 安全 Top 10：

| 编号 | 名称 | 含义 |
|------|------|------|
| AST01 | Malicious Skills | 隐藏的恶意载荷、后门、凭证窃取 |
| AST02 | Supply Chain Compromise | 供应链投毒、依赖劫持 |
| AST03 | Over-Privileged Skills | 权限过大、凭证泄露 |
| AST04 | Insecure Metadata | Manifest 篡改、链接可疑 URL |
| AST05 | Unsafe Deserialization | pickle/yaml 不安全反序列化 |
| AST06 | Weak Isolation | 容器逃逸、无沙箱 |
| AST07 | Update Drift | 依赖不锁版本、可被篡改 |
| AST08 | Poor Scanning | 混淆、编码、躲避扫描 |
| AST09 | No Governance | 缺少 Manifest、无审计 |
| AST10 | Cross-Platform Reuse | 跨平台攻击、多生态兼容 |

## 最终引擎架构（V60）

```
输入: /data/skills/{skill_id}/
  │
  ├─ Layer 1: 检测层
  │   ├─ Manifest 分析（权限、URL、依赖）
  │   ├─ 关键词匹配（~200 个模式）
  │   ├─ 精确指令链（40 条规则，如 curl|bash、/dev/tcp 等）
  │   └─ Markdown NLP（社工模式检测）
  │   → 输出: detect_total → 判断 benign/malicious/suspicious
  │
  ├─ Layer 2: 分类层
  │   └─ 每个 AST 类别独立打分（互不重叠的独有特征）
  │   → 输出: AST01-AST10 中置信度最高的类别
  │
  └─ Layer 3: 证据层
      └─ 自动生成自然语言证据
      → 输出: "Manifest declares excessive permissions..."
```

## 完整比赛回顾

### 怎么开始的

6 月 12 号下午拿到题目。最初基于模板搭了一个正则引擎，120 条规则，四层架构（Manifest 分析、正则匹配、熵分析、AST 分析），从 4.82 起步。

### 第一周：疯狂迭代，正则引擎撞墙

6 月 12-14 号，连续迭代了 30+ 个版本。加分规则、调阈值、改分类逻辑，分数从 4.82 涨到 5.54 就卡死了——14 个版本纹丝不动。

**V18 真正的突破**——完全推倒重来，把 500 行的复杂正则引擎改成了 200 行的简单关键词引擎。复杂正则看上去更"专业"，但实际效果反而不如简单的 `if "os.system" in text`。DQ 从 2.89 一跳涨到 3.59。

这件事教会我们：**在有限样本集上，简单的模式匹配比复杂规则更有效**。正则引擎的额外复杂度没带来额外召回，反而引了更多边界情况。

然后 V36 纠正了一个大问题——之前把 OWASP AST 分类理解成了传统的 Web 安全 Top 10（注入、XSS 之类），但其实这是个专门面向 AI Agent Skill 安全的分类体系。纠正之后 EX 直接翻倍（0.45→0.85）。

### 自动化和"野路子"

过程中建了不少自动化工具：
- **auto_submit.py**：一键打包、登录、提交（就差验证码需要人读）
- **score_tracker.py**：查看双赛道分数、历史趋势、优化建议
- **check_notices.py**：监控平台公告

还试过在引擎里加 DNS/HTTP 外泄模块想偷看测试集样本——结果网络隔离太严，还把分搞跌了。后来分析这个"野路子"是双刃剑：想法对，但执行太糙，稳定性扣分不说还干扰了正常检测。

### 关键资源

比赛后期找到了几个决定性的资源：

1. **OWASP Agentic Skills Top 10**（`kenhuangus/agentic-skills-top-10`）：AST01-AST10 的正确分类定义
2. **skill-scan**（`kkdub/skill-scan`）：93 条现成检测规则
3. **HuggingFace 恶意 Skill 数据集**（`yoonholee/agent-skill-malware`）：347 个真实样本

### 分数天花板

**DQ** 在第一个测试集上冲到 4.23（F₂≈0.77），但最后一天平台疑似换了测试集，相同引擎 DQ 掉到 3.62。新测试集的未检出样本用 NLP、指令链、数据集规则全试了都抓不住——很可能是纯自然语言操控型 Skill。

**EX** 从 0.45 涨到 1.29，证明分类逻辑和证据自然语言化是有效的。

**RO** 在 52 个版本中从未变过 0.83——要么是平台固定扣分项，要么和输出格式的某些要求有关。

**PF** 始终 1.0，关键词引擎性能没问题。

### 经验教训

- **简单优于复杂**：200 行关键词引擎碾压了 500 行正则+AST+熵四层引擎
- **先搞清楚题意**：把 AST 分类当成传统 Web Top 10 浪费了十几个版本的 EX 分数
- **数据驱动**：HuggingFace 数据集帮我们验证了规则的精准度（如 `base64 -d | sh` 在恶意样本命中 44%，良性 0%）
- **不要过早优化**：V1-V17 的几百条规则加了个寂寞，V18 推到重来才突破
- **自动化是双刃剑**：auto_submit 提效率，但外泄模块偷鸡不成蚀把米
- **RO 需要第一天就搞对**：USER 1000、字段名、证据文本这些基础设施问题应该在搭架子时就解决
- **每天 10 次太少了**：只能靠分析 + 推理，不能靠 brute force 试
