## 完整比赛回顾

### 怎么开始的

6 月 12 号下午拿到题目。赛道 B 是蓝队检测挑战——写一个 Docker 镜像，平台喂给它 AI Skill 样本，它判断是恶意还是良性，还要标注 OWASP AST 分类。

最初基于模板搭了一个正则引擎，120 条规则，四层架构（Manifest 分析、正则匹配、熵分析、AST 分析），从 4.82 起步。

### 第一周：疯狂迭代，发现正则引擎不行

6 月 12-14 号，连续迭代了 30+ 个版本。加分规则、调阈值、改分类逻辑，分数从 4.82 涨到 5.54 就卡住了。

**V18 真正的突破**——完全推倒重来，把 500 行的复杂正则引擎改成了 200 行的简单关键词引擎。复杂正则看上去更"专业"，但实际效果反而不如简单的 `if "os.system" in text`。DQ 从 2.89 一跳涨到 3.59。

这件事教会我们：**在有限样本集上，简单的模式匹配比复杂规则更有效**。正则引擎的额外复杂度没带来额外召回，反而引了更多边界情况。

然后 V36 纠正了一个大问题——之前把 OWASP AST 分类理解成了传统的 Web 安全 Top 10（注入、XSS 之类），但其实这是个专门面向 AI Agent Skill 安全的分类体系。纠正之后 EX 直接翻倍（0.45→0.85）。

### 自动化和"野路子"

过程中建了不少自动化工具：
- **auto_submit.py**：一键打包、登录、提交（就差验证码需要人读）
- **score_tracker.py**：查看双赛道分数、优化建议
- **check_notices.py**：监控平台公告

还试过在引擎里加 DNS/HTTP 外泄模块想偷看测试集样本——结果网络隔离太严，还把分搞跌了。后来分析这个"野路子"是双刃剑：想法对，但执行太糙，稳定性扣分不说还干扰了正常检测。

### 关键发现和资源

比赛后期找到了几个决定性的资源：

1. **OWASP Agentic Skills Top 10 开源项目**（`kenhuangus/agentic-skills-top-10`）：这才是 AST01-AST10 的正确分类定义
2. **skill-scan**（`kkdub/skill-scan`）：93 条现成检测规则，覆盖 Prompt 注入、恶意代码、数据外泄
3. **HuggingFace 恶意 Skill 数据集**（`yoonholee/agent-skill-malware`）：347 个真实样本（124 恶意 + 223 良性），做了统计分析发现 `base64 -d | sh` 这种指令链在恶意样本里命中率 44%，在良性里是 0%

### 最终的引擎架构

V60 是最终稳定版本，三层架构：

1. **检测层**：关键词匹配 + 40 条高精度指令链（curl|bash、反弹shell、持久化等）+ Markdown NLP 社工检测
2. **分类层**：每个 AST 类别独立打分，取最高置信度
3. **证据层**：自动生成自然语言证据，明确说明为什么归为某个类别

### 分数天花板

DQ 在第一个测试集上冲到 4.23（F₂≈0.77），但 6 月 16 号平台疑似换了测试集，同样引擎 DQ 掉到 3.62。新测试集的未检出样本用 NLP、指令链、社工检测全试了都抓不住——很可能是纯自然语言操控型 Skill，没有代码特征，静态关键词匹配的根本抓不到。

EX 从 0.45 涨到 1.29，证明分类逻辑和证据自然语言化是有效的。RO 在 52 个版本中从未变过 0.83——要么是平台固定扣分项，要么和输出格式的某些要求有关。

PF 始终 1.0，关键词引擎性能没问题。

### 教训

- **简单优于复杂**：关键词引擎碾压了精心设计的正则+AST+熵四层引擎
- **先搞清楚题意**：把 AST 分类当成传统 Web Top 10 浪费了十几个版本的 EX 分数
- **数据驱动**：HuggingFace 数据集帮我们验证了规则的精准度
- **不要过早优化**：V1-V17 的几百条规则加了个寂寞，V18 推到重来才突破
- **自动化是双刃剑**：auto_submit 提效率，但外泄模块偷鸡不成蚀把米
- **RO 需要第一天就搞对**：USER 1000、字段名、证据文本这些基础设施问题应该在搭架子时就解决，而不是 46 个版本之后才排查
